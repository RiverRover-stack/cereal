# 0005 — Synthetic-only training data: current status and multi-store roadmap

## Status
Accepted (documents current state) / roadmap section Proposed

## Context
The stated plan was "many synthetic stores, generated by Claude, merged into
one master dataframe." Checked against the actual data files:

- `data/processed/master_df_cbrt.csv` — 31,736 rows. This is 40 SKUs × ~793
  usable days (after feature-engineering warm-up) of **one store's** history,
  spanning 2023-01-08 to 2025-12-30 (`feature_metadata.json:7-10`).
- `data/catalogue.csv` — 40 SKU rows with generation parameters
  (`annual_phase`, `annual_amplitude`, `popularity`, `promo_sensitivity`) —
  one store's product catalogue, not multiple stores' catalogues merged.
- `data/components.csv` — the literal trend/weekly/annual/event/promo/noise
  signal decomposition the series was generated from — further confirming
  procedural generation of a single coherent store, not a union of separate
  synthetic businesses.
- **No store or seller identifier column exists in any data file.** There is
  no way to even distinguish "store A's rows" from "store B's rows" because
  there is only one store's rows.

There's no dispute that the data is synthetic (this was always the stated
plan and isn't in question); the finding is narrower and more specific: it's
synthetic data for **one** store, not many merged stores. This matters because
"one store" and "many stores" imply very different claims about what a model
trained on this data has learned — one store's data can teach a model what
*that store's* seasonality and price sensitivity look like; it cannot teach a
model what varies *across* stores, because there's no variation across stores
in the training set to learn from.

## Decision
1. **Record the actual state accurately**: current training/eval data
   (`master_df.csv`, `master_df_cbrt.csv`, `sales_daily.csv`) represents one
   synthetic store. Any claim about "the model generalizes across sellers"
   is unsupported by this data and should not be made until Phase 2 below is
   done.
2. **Roadmap to genuine multi-store synthetic data** (a precondition for the
   ADR-0002 Phase 2 gate, not a commitment to build a global model — see
   ADR-0002):
   - Generate multiple distinct synthetic stores — varied price points,
     category mixes, seasonality shapes, promo cadences — each internally
     coherent the way the current single store is, but different from each
     other in the ways real stores actually differ.
   - Add an explicit store/seller ID column when merging, so held-out-store
     validation (train on stores 1-8, test on store 9) is possible — this is
     the only way to actually test generalization across sellers, synthetic
     or real.
   - Keep per-store data internally clean (no cross-store leakage in rolling
     windows, lags, or splits) — the existing leakage guards
     (`models/features.py`) generalize to this but need a `groupby(store_id)`
     added everywhere they currently only `groupby(sku)`.
3. **Eventual transition to real data** is Phase 3 in the broader roadmap
   (`docs/architecture-review.md` §9) and depends on collecting anonymized,
   opt-in seller data — out of scope for this ADR.

## Consequences
- **Easier**: future work on a global/pretrained model has a concrete,
  checkable data requirement instead of an assumption that today's data
  already supports it.
- **Harder**: generating genuinely varied synthetic stores (not just more days
  of the same store) is real work, and there's currently no generator script
  in the repo for the existing store either — `backend/scripts/generate_sales_data.py`
  produces the *live path's* fixture (`sales_daily.csv`), which is a separate,
  simpler generator from whatever produced `catalogue.csv`/`events.csv`/
  `components.csv` for the offline path; that generator's source was not found
  in this repo during review and may need to be reconstructed or rewritten.
- **Debt**: none new; this documents a gap (single-store data) that already
  existed and was previously undocumented.

## Alternatives considered
- **Treat the current one-store data as "good enough" and proceed to a global
  model anyway.** Rejected — this is exactly the path that produces a model
  which looks validated (good backtest numbers, `docs/forecast-validation.md`)
  but has never actually been tested on a different store's data, which is
  the specific claim "generalizes across sellers" requires.
- **Skip synthetic multi-store data and go straight to real seller data.**
  Not rejected, but sequenced after: real data requires storage and consent
  infrastructure that doesn't exist yet (ADR-0008), while multi-store
  synthetic data can be generated and validated against immediately with the
  existing offline pipeline.
